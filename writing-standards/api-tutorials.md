# API Tutorials

* For OpenAPI `.yml` API reference specs, see [API References](api-references.md).
* For preview features, use the preview badge and danger hint described in [Preview labeling](api-references.md#preview-labeling).
* API tutorials are kept as `.md` files in the `api-references` repository.
* Section titles follow the pattern "How to {do/achieve...}".
* In the page metadata (in the API docs space), specify the icon as `graduation-cap`, and wide layout.

Example:

```
---
icon: graduation-cap
layout:
  width: wide
---
```

## Required API content blocks

* For each workflow using Emporix API endpoints, consider including (as and where appropriate):
  * Purpose – what the endpoint/workflow does
  * Authentication – required token type and scopes
  * Path and method – exact method and endpoint path
  * Required headers – for example `Authorization`, `Content-Type`
  * Request body – required fields, optional fields, defaults
  * Response behavior – success responses and common error responses
  * Results – what changes in the system after success

## `Try it` and example requests

* Add `Try it` request examples to demonstrate behavior and request construction.
* Add a success hint – either use reusable content from GitBook app (API Documentation space), or include this snippet:

```
{% include "../../.gitbook/includes/example-hint-text.md" %}
```

* Add a link to the endpoint.
* Add a complete `curl` request in a `bash` code block.
* Prefer realistic values over generic placeholders where safe.
* Include at least one error example when failure is common or costly.

Example in GitBook:

Example:

```bash
curl -i -X POST \
  'https://api.emporix.io/fee/{tenant}/fees' \
  -H 'Authorization: Bearer {{OAUTH2_ACCESS_TOKEN}}' \
  -H 'Content-Language: string' \
  -H 'Content-Type: application/json' \
  -d '{
    "name": {
      "de": "Sixpack PET",
      "en": "Sixpack PET"
    },
    "code": "apple_picking_fee",
    "feeType": "ABSOLUTE",
    "siteCode": "main",
    "feeAbsolute": {
      "amount": 3.5,
      "currency": "EUR"
    },
    "active": true,
    "taxable": false
  }'
```

## Token placeholders

For tokens in example requests, use:

* Service OAuth2 token: `{{OAUTH2_ACCESS_TOKEN}}`
* Customer token: `{{CUSTOMER_ACCESS_TOKEN}}`
* Anonymous token: `{{ANONYMOUS_ACCESS_TOKEN}}`

## Endpoint links and naming

* Link endpoints by path-based URLs as generated by GitBook, not by operationIDs.
* Keep display labels aligned with API reference naming.
* In changelog content, use gerund form for endpoint link text (for example, "Retrieving cart by ID").

## Product and platform terms

* Use official service names consistently (for example, Cart Service, Product Service).
* Keep code identifiers exactly as implemented (`tenant`, `siteCode`).
* Use backticks for endpoint paths, parameter names, setting names, and code-related values.

## Behavior clarity rules

* Call out rate limits, timeouts, and retries if applicable.
* Mention eventual consistency if users may observe delayed data updates.

## Deprecation and compatibility

* Mark deprecated endpoints with the `deprecated: true` and link to migration targets (where applicable).
* Provide migration steps when behavior or schemas have changed.
*   When a whole service becomes deprecated:

    * Add a tag to the service tutorial, and api-reference pages in the frontmatter.

    Example:

    ```
    tags:
    - deprecated
    ```

    * Add deprecation note to service parnet page.

    Example:

    * Add deprecation to all service endpoints.
